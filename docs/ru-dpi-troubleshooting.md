# Обход ТСПУ РФ: диагностика и лечение

Практический разбор «подключается, но интернета нет» на российских сетях.
Основано на живых тестах из РФ + проверке источников (net4people #490/#516/#546 и др.).

## Главный принцип: критерий успеха — НЕ «подключилось»

Reality идеально маскирует TLS-хендшейк. Поэтому «коннект есть / порт пингуется» **ничего не
доказывает** — поток может быть мёртв. **Единственный честный критерий — крупная загрузка
(> 1 МБ) через туннель без обрыва.** Тестируй скачиванием большого файла (например
`https://speed.cloudflare.com/__down?bytes=5000000`) или спидтестом, а не «открылся ли сайт».

## Две болезни (лечатся по-разному) — сначала диагноз

### (A) Заморозка TCP после ~16 КБ
Хендшейк проходит, мелочь грузится, крупное виснет (поток сервер→клиент глохнет без RST).
**Лечится:** сменой транспорта (gRPC; XHTTP в режиме `packet-up`), фрагментацией ClientHello,
иногда другим портом.

### (B) Мобильный whitelist по SNI / IP
Режет уже на хендшейке/первом пакете, потому что иностранный IP + не-разрешённый SNI не в белом
списке оператора. **Смена транспорта НЕ помогает.** Нужен либо разрешённый RU-SNI (спасает не
всегда), либо **РФ-релей с «белым» IP** (см. [architecture.md](architecture.md)).

> Дата-центровый DPI мягче абонентского/мобильного: «работает с VPS в РФ» ≠ «работает на
> домашнем/мобильном». Проверяй на реальной проблемной сети.

## Диагностика на устройстве (10 минут)

1. **UDP-проба.** Подключи AmneziaWG (UDP). Работает → UDP проходит, проблема в TCP (склоняемся
   к A). Не работает → вероятен whitelist/жёсткий UDP-drop (ближе к B).
2. **gRPC-проба.** Импортируй gRPC-конфиг, скачай > 1 МБ через туннель. Качает → gRPC пробивает,
   делай основным.
3. **Тест «мелкое vs крупное».** Мелочь идёт, крупное виснет на ~16 КБ → болезнь A. Совсем ничего,
   даже хендшейка → ближе к B.
4. **Сопоставь сети.** Прогони то же на домашнем проводном и на мобильном — DPI у них разный.

## Что лечит (по убыванию полезности, без своего домена)

1. **gRPC + Reality** — основной рычаг против A. В этом проекте основной транспорт (`:2053`).
2. **TLS-фрагментация ClientHello** (клиент, без изменений сервера) — против блокировки по SNI
   на хендшейке. Привязывается к Reality-outbound через `sockopt.dialerProxy` → freedom+fragment:
   ```json
   "streamSettings": { ..., "sockopt": { "dialerProxy": "frag" } }
   // отдельный outbound:
   { "protocol": "freedom", "tag": "frag",
     "settings": { "fragment": { "packets": "tlshello", "length": "100-200", "interval": "10-20" } } }
   ```
   Не помогает — ужимай `length` (`100-200`→`40-80`→`10-20`). Не ставь `interval:"0"` с `tlshello`.
   Генерируется флагом `--frag` у `bin/gen-client-config.py`.
3. **XHTTP + Reality** (доп. ставка, не серебряная пуля) — против A тестируй режим **`packet-up`**
   (дробит download на много коротких коннектов и сбрасывает поконнектный лимит 16 КБ),
   НЕ `stream-one`. Явно указывай `mode` и на сервере, и на клиенте.
4. **Разрешённый RU-SNI** на whitelist-инбаунде (моб. РФ) — держи его на `443`, не на высоком порту.
5. **РФ-релей с белым IP** — единственное, что обходит И заморозку, И whitelist одновременно.

## Развенчанные мифы (не трать на это время)

- **«XHTTP `stream-one` спасёт от заморозки 16 КБ»** — нет, он держит весь download на одном
  соединении и упирается в тот же лимит. Против заморозки — `packet-up`. И даже он помогает не всем.
- **«Пустой SNI / убрать fingerprint починит»** — для Reality невозможно: он работает на uTLS,
  пустой fp = ошибка коннекта.
- **«Уйти с 443 на высокий порт»** — для мобильного whitelist контрпродуктивно: там настоящий
  HTTPS живёт на 443, TLS на другом порту — аномалия. (Для домашнего проводного иногда помогает.)
- **«SNI российского банка (sberbank.ru и т. п.)»** — худший выбор: RU-сайт на иностранном IP =
  максимальное несоответствие IP↔SNI, ТСПУ это ловит.
- **«ML-DSA / пост-квант усложнит блокировку»** — нет, это про будущий квантовый MITM, к обходу
  отношения не имеет; плюс может сломать коннект на неподдерживающем клиенте.
- **«Reality можно с WebSocket/HTTPUpgrade»** — нельзя: Reality сочетается только с RAW(TCP),
  gRPC, XHTTP.

## «Ошибка геоданных» в Happ

Это провал внутренней гео-/connectivity-проверки клиента — может означать и мёртвый канал, и
просто обрезанные гео-базы. Диагностически: включи проксирование всего и открой любой сайт;
не грузится → проблема в канале (см. выше), а не в гео. Держи `DomainStrategy=IPIfNonMatch`,
DNS по DoH через прокси.
